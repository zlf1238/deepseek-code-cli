# 工具清理与质量提升 — 决策记录

对本项目内置工具的全面审查与清理：废弃冗余工具、修正误导性描述、修复测试 bug、重构耦合代码、提升容错能力。

---

## 一、问题背景

### 1.1 触发点

用户询问"除了 ask_choice 弹窗还有哪些弹框"，触发了对项目工具集的全貌审视。`grep` 返回空结果暴露了内置 grep 的 BRE 正则方言问题，进而引发对全部工具描述准确性、实现质量和冗余度的系统审查。

### 1.2 识别的问题全景

| # | 问题 | 严重度 | 类型 |
|---|------|:------:|------|
| 1 | `search_files` 与 `glob` 功能重叠，且经验证明不稳定 | 中 | 冗余 |
| 2 | `grep` 工具描述未注明使用 BRE 基本正则，`\|` 被当作字面量 | 中 | 描述误导 |
| 3 | `search_files` 描述反向推荐优于 `glob`，与实战经验矛盾 | 中 | 描述误导 |
| 4 | `read-handler` snippet startLine 硬编码为 1，测试失败 | 低 | Bug |
| 5 | `multi_edit` 与 `edit-handler` 两套独立实现，裸 fs 调用 | 中 | 技术债 |
| 6 | `multi_edit` 描述称"原子操作"，实际失败继续执行 | 低 | 描述误导 |
| 7 | `require_confirmation` 参数定义存在但 handler 从未使用 | 低 | 死代码 |
| 8 | `ask_choice` 与 `AskUserQuestion` 两套独立 handler | 中 | 冗余 |
| 9 | GitNexus `shouldReindex` 检查目录 mtime（WSL2 9p 不可靠），无降级 | 高 | 容错缺陷 |

## 二、决策思路

### 2.1 search_files：废弃 vs 保留

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A. 废弃 | 删除工具和处理函数，引导 AI 用 glob | 净减代码，消除决策混淆 | 丢失 find 命令的大目录搜索性能 |
| B. 保留 + 降级描述 | 保留工具但修改描述建议优先用 glob | 不破坏兼容性 | 工具仍在列表中，AI 仍需在两个工具间选择 |
| C. 合并到 glob | 让 search_files 委托到 glob handler | 保留 API 兼容 | glob 模式与文件名子串语义不同，转换有损耗 |

**选 A**。理由：
- prompt 和 AGENTS.md 已明确说它不如 glob 稳定
- 唯一差异（自动排除 node_modules）glob 也可通过路径模式实现
- 减少 AI 的工具选择负担，提升决策效率

### 2.2 grep 描述修正：改源码 vs 改描述 vs 只写文档

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A. 改源码（加 `-E`） | 切到扩展正则，`\|` 自然支持 OR | 一劳永逸 | 行为变更，`+?(){}` 语义全部变化，需全套回归 |
| B. 改工具描述 | 在 pattern 参数描述中注明 BRE 限制 | 零风险，AI 直接看到 | 不改变工具实际行为 |
| C. 只写 AGENTS.md | 在实战经验中记录 | 成本最低 | AI 未必在工具调用时查阅经验条目 |

**选 B**。理由：
- 工具描述是 AI 每次调用前都会看到的信息源，从源头纠正
- 方案 A 风险过高（BRE→ERE 语义变化影响所有正则调用方）

### 2.3 multi_edit 重构：废弃 vs 委托 vs 复用 file-utils

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A. 废弃 | 删除 multi_edit，AI 并行调多个 edit | 消除重复代码 | Fixer 子智能体核心工具，多文件需多次 API 往返 |
| B. 委托到 edit | multi_edit 逐条调用 handleEditTool | 获得所有增全特性 | handleEditTool 要求先 read + 有 fileState，与批量语义冲突 |
| C. 复用 file-utils | 用 readTextFileWithMetadata/writeTextFile 替代裸 fs | 务实，不改架构 | 仍独立于 edit，不支持 snippet/candidate 等高级特性 |

**选 C**。理由：
- 方案 A 会严重降低 Fixer 效率（子智能体 system prompt 明确推荐 multi_edit）
- 方案 B 的 "先 read 再 edit" 前置条件与批量编辑场景天然矛盾
- 方案 C 用最小的改动获得最大收益：编码检测、CRLF 保持、文件状态记录

### 2.4 ask_choice 合并：委托适配 vs 完全删除

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A. 委托适配 | executor 中转换为 AskUserQuestion 格式后委托 | 工具名和 schema 不变，LLM 无感知 | allowCustom:false 时 UI 仍显示 Other |
| B. 完全删除 | 删除工具，LLM 只用 AskUserQuestion | 最大简化 | LLM 失去简洁参数格式，可能降低调用意愿 |

**选 A**。理由：
- 对 LLM 完全透明（工具名和 schema 不变）
- 仅需改 executor.ts 的注册逻辑 + 删除 ask-choice-handler.ts
- 改动面最小（不改 prompt、UI、session 层）

### 2.5 GitNexus 容错：改 shouldReindex + 降级

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A. 修 shouldReindex | 改为检查 meta.json 的 indexedAt 字段 | 解决 WSL2 9p 根因 | 不解决索引失败后的服务不可用 |
| B. A + 降级逻辑 | 重新索引失败但已有有效数据时继续使用 | 5 个工具在索引维护期间仍可用 | 可能使用过期索引数据 |
| C. 全局安装 gitnexus | 绕过 npx 依赖安装问题 | 根除安装失败 | 环境依赖，不是代码层面解决 |

**选 B**。理由：
- 方案 A 只修复了"误触发重索引"，未解决"重索引失败导致服务中断"
- 方案 C 需用户手动操作，不是代码解决方案
- 过期索引 > 完全不可用——降级是最务实的容错策略

### 3.5 GitNexus 容错改进

```typescript
// shouldReindex: 目录 mtime → meta.json indexedAt
function shouldReindex(projectRoot: string): boolean {
  const metaPath = path.join(getGitnexusDir(projectRoot), "meta.json");
  const meta = JSON.parse(fs.readFileSync(metaPath, "utf8"));
  return Date.now() - new Date(meta.indexedAt).getTime() > MAX_INDEX_AGE_MS;
}

// withIndex: 添加降级路径
async function withIndex(context, fn) {
  const indexResult = await ensureIndex(context.projectRoot, context);
  if (!indexResult.ok) {
    if (isIndexed(context.projectRoot)) {
      return fn();  // 降级使用现有索引
    }
    return { ok: false, name: "", error: indexResult.error };
  }
  return fn();
}
```

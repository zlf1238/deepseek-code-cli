## 实战经验手册

> 核心教训。保留规则均为无法自动化的经验判断与思维模型。
> 已归档规则见项目根目录 `AGENTS-ARCHIVE.md`。

- **分析问题先分"设计行为"与"代码缺陷"** —— 出现错误先追问：这是有意的安全守卫还是真正的 bug？追踪完整调用链（报错点 → 上层调用 → 参数来源）找到根因。
- **复用已有工具函数** —— 修复前先 grep 项目中是否已有类似功能的函数，避免重复造轮子。已有函数未被所有应该使用它的地方调用时，补充调用而非另起炉灶。
- **理解参数传递链** —— 工具调用失败时，阅读底层库的 `resolveRepo`/`callTool` 等核心方法，理解 MCP 参数传递机制。很多"环境配置类"错误的根因是缺少某个必需的参数。
- **用户表述模糊时先确认** —— 有歧义时用一句话确认理解后再展开方案。宁可多花一轮确认，也不要花两轮纠偏。
- **选定方案后先描述用户体验闭环** —— 实现前描述完整体验："选了这个方案后，你会看到 XXX，需要额外操作 YYY 才能 ZZZ。"确保用户在知情前提下决策——尤其当方案有功能性取舍（如截断 vs 完整渲染）。
- **`run_background` 任务未出结果前不回复"完成"** —— 连续 3 次 job_output 触发循环防护后，应切到 list_jobs 查看任务状态。提交 git 前必须确认所有相关测试完成。

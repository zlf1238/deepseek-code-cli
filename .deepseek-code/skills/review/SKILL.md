# Review Skill

## Quick Start

当用户要求你审查代码时，按以下流程操作：

1. **获取 diff** — `bash("git diff")` 或 `bash("git diff origin/main...HEAD")` 获取变更内容
2. **阅读被修改的文件** — 使用 `read` 理解完整上下文，而非仅看 diff 片段
3. **按维度审查** — 逐项检查以下维度
4. **输出审查报告** — 结构化的 verdict + 逐项 issue

## 审查维度

### 1. 正确性
- 代码逻辑是否按预期工作？
- 边界条件是否处理？（null, 空数组, 0, 缺失参数）
- 类型是否安全？（any 滥用, 缺少类型守卫）

### 2. 安全性
- 是否引入了注入漏洞？（SQL, shell, path traversal）
- 敏感信息是否硬编码或日志泄露？
- 输入是否校验？（URL, 文件路径, 用户输入）

### 3. 错误处理
- 异常路径是否被捕获？
- 错误信息是否有用？
- 失败后会留下不一致的状态吗？

### 4. 测试
- 新增功能是否有测试？
- 修复的 bug 是否有回归测试？
- 现有测试是否仍能通过？

### 5. 架构 / 一致性
- 新代码是否遵循项目现有模式？
- 是否引入了不必要的依赖？
- 模块边界是否合理？

## 输出格式

```markdown
# 审查报告

**Verdict**: ✅ 通过 / ⚠️ 建议修改 / ❌ 需要修改

## Issues

### 1. [严重程度: high/medium/low] [标题]
- **文件**: path/to/file.ts:42-45
- **问题**: 具体描述
- **建议**: 修复方案
- **理由**: 为什么需要改

### 2. ...
```

## 注意事项

- 不修改代码——只审查，不编辑
- 对不确定的结论使用 "可能" / "建议确认" 等措辞
- 如果 diff 为空，提示用户确认分支状态
- severity: high = 必须修，medium = 建议修，low = 可选优化
